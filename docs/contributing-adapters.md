# Contributing New Adapters

> Want to add support for a new source environment, stack, or workload pattern? This guide walks you through it.

## 🚨 Critical first rule — where to edit

This is a **monorepo** that uses a build artifact pattern. Always edit the canonical files at the **monorepo root** — never the copy under `packages/azure-migration-squad/templates/`.

| You're editing... | ✅ Edit here | ❌ NEVER touch |
|-------------------|--------------|---------------|
| A skill (adapter, decision tree, capability matrix) | `.github/skills/*.md` | `packages/azure-migration-squad/templates/github/skills/` |
| A prompt | `.github/prompts/*.prompt.md` | `packages/azure-migration-squad/templates/github/prompts/` |
| A chatmode | `.github/chatmodes/*.chatmode.md` | `packages/azure-migration-squad/templates/github/chatmodes/` |
| An agent charter | `.squad/agents/<name>/charter.md` | `packages/azure-migration-squad/templates/squad/agents/` |
| AGENTS.md or copilot-instructions.md | root or `.github/copilot-instructions.md` | `templates/AGENTS.md` etc. |

**Why?** The `templates/` folder is regenerated by `npm run sync` before every publish, in CI on every PR, and by the `prepack` hook. Edits there will be silently overwritten. CI also runs `scripts/check-templates-not-edited.mjs` and fails the build if it detects manual edits to `templates/`.

After editing a canonical file, **run `npm run sync`** (from `packages/azure-migration-squad/`) so the package mirror catches up. Commit both files. Done.

If you ever accidentally edit `templates/` and the CI guard catches it, just move your changes to the canonical location and re-run sync. No data loss.

---

The Azure Migration Squad is designed to be **universally extensible** along three axes:

| Axis | Files | What you're adding |
|------|-------|-------------------|
| `source-*` | `.github/skills/source-<name>.md` | Where the application currently lives (e.g. `source-bitbucket-cloud`, `source-heroku`) |
| `stack-*` | `.github/skills/stack-<name>.md` | What the application is built in (e.g. `stack-elixir`, `stack-haskell`) |
| `workload-*` | `.github/skills/workload-<name>.md` | What architectural pattern it follows (e.g. `workload-realtime-streaming`, `workload-ml-training`) |

The Discovery Engineer auto-discovers your adapter as soon as the file lands in `.github/skills/` — no manual registration needed.

## Quick start

```bash
# 1. Fork + clone
git clone https://github.com/<your-fork>/GHCP-PromptMigration
cd GHCP-PromptMigration
npm install

# 2. Pick a name following the convention
#    source-X / stack-X / workload-X — lowercase, hyphen-separated

# 3. Copy an existing adapter as a starting template
cp .github/skills/source-github-repo.md .github/skills/source-bitbucket-cloud.md
# (or stack-python.md / workload-webapp.md depending on axis)

# 4. Edit the file (see template structure below)

# 5. Run validation
npm run sync --workspace=@robertoborges/azure-migration-squad
npm run build --workspace=@robertoborges/azure-migration-squad
npm run lint --workspace=@robertoborges/azure-migration-squad
node scripts/validate-plugin-manifest.mjs

# 6. If green, regenerate plugin manifest (it auto-includes your skill)
npm run build:plugin-manifest

# 7. Open a PR
```

## Adapter template structure

Every adapter follows the same shape. Use one of these proven templates as a starting point:

- **source adapter** — see [`.github/skills/source-github-repo.md`](../.github/skills/source-github-repo.md) (deep example) or [`source-gcp.md`](../.github/skills/source-gcp.md) (stub example)
- **stack adapter** — see [`.github/skills/stack-python.md`](../.github/skills/stack-python.md) (deep) or [`stack-ruby.md`](../.github/skills/stack-ruby.md) (stub)
- **workload adapter** — see [`.github/skills/workload-webapp.md`](../.github/skills/workload-webapp.md) (deep) or [`workload-serverless.md`](../.github/skills/workload-serverless.md) (stub)

Required sections (in order):

1. **Title + 1-line summary** — `# Skill: <axis> Adapter — <Name>`
2. **When to use** — what Capability Matrix value triggers this adapter
3. **Inputs** — what the Discovery Engineer needs to give you
4. **Probes** — the steps the Discovery Engineer takes to gather evidence (CLI commands, file lookups, API calls)
5. **Output Evidence** — the YAML block that gets added to `Capability-Matrix.yaml`
6. **Migration Constraints / Risks** — high-impact things to surface to the Architect
7. **Target Azure Mapping** — a table mapping source-shape → Azure candidate
8. **Anti-Patterns** — what NOT to do
9. **Output Checklist** — bulleted criteria the Discovery Engineer self-checks before finishing

## What makes a good adapter

✅ **Cite evidence.** Every classification points to a file path, command output, or quoted user statement. No prose-only evidence.

✅ **Confidence labels.** Output specifies `evidence_confidence: high | medium | low` and ties low confidence to follow-up probes.

✅ **Stay in your lane.** A `source-*` adapter doesn't classify the stack — that's `stack-*`'s job. Cross-reference with `(handoff to stack-detection)`.

✅ **Realistic Azure candidates only.** Don't promise PaaS for stacks that fundamentally need IaaS (e.g., Oracle Forms doesn't run on App Service).

✅ **Cover both deep and shallow paths.** Mention "stub adapter" if you're providing only basic recognition + escalation guidance (acceptable for long-tail).

❌ **Don't generate code or IaC in skills.** Skills are for guidance. Code generation is a Phase 2/3 prompt's job.

❌ **Don't bypass the Discovery → Architect handoff.** Skills propose; the Architect decides.

❌ **Don't hardcode SKUs or regions.** Skills propose *services*; the Architect picks SKU + region per engagement.

## Schema requirements

If your adapter introduces new values for enum fields in `Capability-Matrix.yaml`, you must also update:

| Schema | What to update |
|--------|---------------|
| `packages/azure-migration-squad/schemas/capability-matrix.schema.json` | Add your new value to the `enum` array for the relevant field (`source.primary_adapter`, `stack.primary_stack`, `workload.primary_pattern`) |
| `.github/skills/stack-detection.md` | If a new stack: add detection signals to the table |
| `.github/skills/migration-strategy-decision-tree.md` | If your adapter changes a branch outcome: update the relevant branch |
| `.squad/routing.md` | If the adapter implies new specialist routing: update the routing tables |

The CI's `scripts/validate-plugin-manifest.mjs` will fail the build if `Capability-Matrix.yaml` enum values are out of sync.

## Testing your adapter

There's no automated test for skill content (yet — coming in Wave E's Evaluator-driven eval suite). Manual testing:

```bash
# 1. Build the package locally
cd packages/azure-migration-squad
npm run build

# 2. Install into a test repo
mkdir /tmp/test-adapter && cd /tmp/test-adapter
git init && mkdir .squad && echo "# test squad" > .squad/team.md
node /path/to/GHCP-PromptMigration/packages/azure-migration-squad/bin/cli.js init

# 3. Verify your file landed
ls .github/skills/ | grep <your-adapter-name>

# 4. Open Copilot Chat and try a discovery:
/assess-any-application
# Provide answers that should trigger your adapter
# Verify the Discovery Engineer loads it
```

## PR checklist

Before opening your PR, verify:

- [ ] File naming follows the convention (`source-*.md` / `stack-*.md` / `workload-*.md`)
- [ ] All required sections present (see template structure above)
- [ ] Evidence cells cite specific files/commands
- [ ] Confidence labels in output schema
- [ ] No hardcoded SKUs or regions
- [ ] If enum values changed: schema updated, decision tree updated, routing updated
- [ ] `npm run sync` succeeds
- [ ] `npm run build --workspace=@robertoborges/azure-migration-squad` succeeds
- [ ] `npm run lint --workspace=@robertoborges/azure-migration-squad` succeeds
- [ ] `npm test --workspace=@robertoborges/azure-migration-squad` succeeds (7/7 tests)
- [ ] `npm run build:plugin-manifest && node scripts/validate-plugin-manifest.mjs` succeeds
- [ ] Manual install + Copilot Chat smoke test completed
- [ ] CHANGELOG entry added under `## [Unreleased]` (Changesets coming in Wave E will automate this)

## Naming conventions

| Convention | Example |
|------------|---------|
| File names | lowercase, hyphenated, `.md` (`source-bitbucket-cloud.md`) |
| Skill IDs | match file name without extension |
| Capability Matrix enum values | exact file name (without `.md`) |
| Decision tree branch references | `Branch N (descriptor): TRIGGERED — <reason>` |
